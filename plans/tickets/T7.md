## 1. Ticket ID

`T7`

## 2. Goal

Unify workspace, versioning, publish, and migrations into one canonical path.

## 3. Why this ticket exists

The system is not operationally safe if workspace drafts, publish artifacts, repositories, and migrations describe different realities.

## 4. In scope

- Define one canonical flow from workspace draft to published release.
- Remove or explicitly deprecate divergent legacy in-memory paths.
- Align workspace draft model and publish model.
- Ensure migrations, repository implementations, and schemas agree.
- Add migration tests and publish integration tests.

## 5. Out of scope

- Prompt tuning or prompt shape redesign beyond using the `T6` result.
- Emotional runtime changes.
- Ranker gate work, rollout, or legacy heuristic deletion.
- Later tickets that MUST NOT start yet: `T8`, `T9`.

## 6. Required code areas

- `src/lib/versioning/*`
- Workspace repository paths
- Prompt bundle repository paths
- Phase graph repository paths
- Migration files
- Release or publish path files
- Publish and migration tests under `tests/`

## 7. Forbidden code areas

- CoE extractor, integrator, and emotional runtime logic files
- Ranker gate and eval tuning files
- Legacy heuristic cleanup files
- Prompt wording or tuning changes

## 8. Acceptance criteria

- Workspace draft to publish works through one explainable canonical flow.
- Prompt bundle and phase graph versions are saved correctly.
- A fresh database works after migrations.
- Any compatibility path is documented clearly.

## 9. Required tests

- Migration tests against a fresh database
- Repository and schema consistency tests for workspace and publish paths
- Publish integration tests covering prompt bundle and phase graph version persistence

## 10. Stop condition

- Stop if the canonical publish flow cannot be described as a single path before coding.
- Stop if migrations and repository behavior disagree.
- Stop if the work starts to retune prompt content or ranker behavior.

## 11. Risks / notes

- The master plan does not define a new artifact schema beyond "canonical flow"; keep changes limited to making one existing path authoritative rather than adding new product surfaces.
- If a legacy in-memory path must remain temporarily, mark it deprecated explicitly and document its boundary.

## 12. Handoff prompt template

```text
Work only on T7 using plans/tickets/T7.md and COE_PAD_REPAIR_PLAN.md.
Define one canonical workspace-to-publish path, prove it with migration and publish tests, and stop if the work starts changing emotional runtime, prompt tuning, or ranker behavior.
```
