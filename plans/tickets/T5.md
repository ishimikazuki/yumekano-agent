## 1. Ticket ID

`T5`

## 2. Goal

Make draft and sandbox chat stateful and aligned with production behavior.

## 3. Why this ticket exists

The migration is incomplete if sandbox or draft chat resets internal state each turn or uses a different emotional update path than production.

## 4. In scope

- Carry sandbox session state across turns for PAD, `trust`, `affinity`, `conflict`, `intimacyReadiness`, open threads, current phase, and relevant working-memory-derived state.
- Ensure reset happens only on explicit reset behavior.
- Reuse the same CoE to integrator path as production.
- Keep sandbox data isolated from production data while keeping sandbox state persistent within its own session.
- Add multi-turn carry-over and explicit reset tests.
- Add migrations only if persistence changes require them.

## 5. Out of scope

- Production path redesign.
- Prompt source-of-truth unification.
- Publish, versioning, ranker gate work, rollout, or legacy cleanup.
- Later tickets that MUST NOT start yet: `T6`, `T7`, `T8`, `T9`.

## 6. Required code areas

- `src/mastra/workflows/draft-chat-turn.ts`
- Workspace or sandbox repository paths
- Sandbox state persistence paths
- Migrations only if needed for sandbox persistence
- Draft and sandbox tests under `tests/`

## 7. Forbidden code areas

- Production publish and versioning flow files
- Prompt schema and prompt storage cleanup files
- Ranker gate or full eval files
- Legacy heuristic cleanup files

## 8. Acceptance criteria

- A two-turn carry-over test passes.
- A five-turn progression test passes.
- Draft or sandbox phase, PAD, and pair state continue across turns within the same session.
- An explicit reset test passes.

## 9. Required tests

- Two-turn carry-over test
- Five-turn progression test
- Explicit reset test
- Isolation test proving sandbox data does not overwrite production data

## 10. Stop condition

- Stop if the change needs prompt bundle redesign to work.
- Stop if the persistence model for sandbox versus production cannot be described clearly before coding.
- Stop if the work starts to change publish or release flow.

## 11. Risks / notes

- The master plan calls for a "relevant working memory summary" but does not define a new schema for it; reuse the current persisted summary shape if one exists and record ambiguity if it does not.
- If a migration is needed, keep it sandbox-scoped and do not combine it with `T7` versioning work.
