## 1. Ticket ID

`T1`

## 2. Goal

Define the CoE and appraisal contracts in schemas and types without changing runtime behavior.

## 3. Why this ticket exists

The migration cannot proceed safely until the repository has a typed, validated contract for what CoE returns and what the downstream integrator consumes.

## 4. In scope

- Add or finalize Zod schemas and TypeScript types for `CoEEvidence`, `RelationalAppraisal`, `EmotionStateDelta`, and related trace payloads.
- Document explicit numeric ranges for each contract field in code and tests.
- Add parser tests for invalid, partial, and malformed model output.
- Keep the old heuristic appraisal path alive behind an adapter.

## 5. Out of scope

- Production or draft workflow wiring.
- CoE agent implementation.
- PAD or pair-metric integration logic.
- Prompt bundle consolidation, versioning changes, ranker changes, or legacy cleanup.
- Later tickets that MUST NOT start yet: `T2`, `T3`, `T4`, `T5`, `T6`, `T7`, `T8`, `T9`.

## 6. Required code areas

- `src/lib/schemas/*`
- `src/lib/types/*` only if a new type location is necessary
- Existing adapter paths that keep heuristic appraisal available
- Parser and schema tests under `tests/`

## 7. Forbidden code areas

- `src/mastra/workflows/chat-turn.ts`
- `src/mastra/workflows/draft-chat-turn.ts`
- Semantic changes in `src/mastra/agents/*`
- Integrator or PAD update logic files
- Prompt, publish, and versioning flow files

## 8. Acceptance criteria

- The contracts are fixed in schemas, types, and parser tests.
- Malformed or partial model output is rejected or downgraded safely.
- The old heuristic path still works through an adapter.
- The new contracts are not wired into production yet.

## 9. Required tests

- Parser tests for invalid payloads
- Parser tests for partial payloads
- Parser tests for malformed payloads
- Contract tests proving safe fallback to the old path when parsing fails

## 10. Stop condition

- Stop if defining numeric ranges would require a new product decision not present in the current emotion model.
- Stop if making the contracts compile would require runtime workflow wiring.
- Stop if the work starts to implement `T2+`.

## 11. Risks / notes

- The master plan requires explicit ranges but does not enumerate every exact range; keep any unresolved range question visible in code comments or notes rather than inventing product semantics silently.
- If an existing schema already overlaps, consolidate it instead of creating a second contract source.

Work only on T1 using plans/tickets/T1.md and COE_PAD_REPAIR_PLAN.md.
Add or finalize the CoE contracts, write parser tests first, keep the old heuristic path behind an adapter, and do not wire anything into production or draft runtime yet.
