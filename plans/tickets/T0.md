## 1. Ticket ID

`T0`

## 2. Goal

Freeze the current behavior and create a regression and eval harness before any runtime refactor.

## 3. Why this ticket exists

Later tickets are unsafe until the repository can show what is currently broken, keep that failure visible, and rerun the same cases after each change.

## 4. In scope

- Standardize the baseline commands: `npm run test:unit`, `npm run test:integration`, `npm run test:migrations`, `npm run eval:emotion`, `npm run eval:full`.
- Add or normalize at least 10 fixtures for the required emotion and relationship cases.
- Ensure each fixture can evaluate PAD delta band, pair metrics delta band, and CoE reason fields.
- Save a markdown report of the current failing baseline under `tests/` or the existing eval report location.
- Expose only minimal non-behavioral test seams if the current runtime is hard to probe.

## 5. Out of scope

- Any runtime refactor or behavior change beyond test seam exposure.
- Any rewrite of appraisal logic.
- Any CoE contract design, CoE agent work, integrator work, production wiring, sandbox persistence work, prompt consolidation, versioning work, ranker gating, or legacy cleanup.
- Later tickets that MUST NOT start yet: `T1`, `T2`, `T3`, `T4`, `T5`, `T6`, `T7`, `T8`, `T9`.

## 6. Required code areas

- `package.json`
- `tests/`
- `tests/fixtures/`
- Existing eval harness location such as `tests/evals/`
- `vitest.config.*` or the repository's equivalent test config
- CI configuration files if command wiring is missing

## 7. Forbidden code areas

- Semantic edits in runtime behavior files such as `src/lib/rules/appraisal.ts`, `src/lib/rules/pad.ts`, `src/mastra/workflows/chat-turn.ts`, and `src/mastra/workflows/draft-chat-turn.ts`
- Publish, versioning, prompt storage, or ranker logic files

## 8. Acceptance criteria

- The standardized test and eval commands run locally.
- At least 10 fixtures exist for the cases listed in the master plan.
- The baseline report shows that the current state still has visible failures.
- The report is stored under `tests/` or the existing eval report area.

## 9. Required tests

- Command contract tests for the standardized scripts
- Fixture coverage tests for the 10 required cases
- Assertions for PAD delta band, pair metrics delta band, and CoE reason fields per fixture
- Baseline report generation or snapshot test proving failures are recorded

## 10. Stop condition

- Stop if the harness requires semantic runtime changes to pass.
- Stop if the needed eval location is unclear and using it would create a second report system.
- Stop if the work starts to define `T1+` contracts or runtime behavior.

## 11. Risks / notes

- The master plan names `evals/`, but this repository already uses `tests/evals/`; use the existing location rather than creating a parallel harness.
- If test command names already exist with different behavior, normalize carefully and keep the change limited to command wiring and baseline coverage.

## 12. Handoff prompt template

```text
Work only on T0 using plans/tickets/T0.md and COE_PAD_REPAIR_PLAN.md.
Read the repository docs listed in the ticket, write failing tests first, do not change runtime behavior except minimal test seams, and stop if the work would enter T1 or later.
```
